## 1. 平台与运行时(v1 · 已读 Luffa 官方文档)

**核心结论**:Luffa SuperBox = **微信小程序克隆架构**。WXML(模板)+ WXSS(样式)+ JS(逻辑),逻辑层和渲染层双线程分离,API 命名空间是 `wx.*`。这意味着**所有兼容微信小程序的工具链(Taro / uni-app)直接能用**——这是项目最大的好消息。

来源:
- [Introduction](https://luffa.im/SuperBox/docs/hk/miniProDevelopmentGuide/Introduction.html)
- [Framework](https://luffa.im/SuperBox/docs/hk/miniProDevelopmentGuide/Guideline/MiniProFramework/introduce.html)
- [运行环境](https://luffa.im/SuperBox/docs/hk/miniProDevelopmentGuide/Guideline/UnderstandingMiniPro/env.html)
- [Relay 分享 API](https://luffa.im/SuperBox/docs/hk/API/relay.html)
- [开放 API](https://luffa.im/SuperBox/docs/hk/API/openapi.html)

### 1.1 已确认的关键能力

| 能力 | 实现 | 影响 |
|---|---|---|
| 开发模型 | WXML + WXSS + JS,逻辑/渲染双线程,Page({ data, methods })  + `{{var}}` 数据绑定 + `bindtap` 事件 | §2 前端栈选 Taro,prototype.html 视觉照搬 |
| Web API | DOM/BOM **不可用**;Canvas / Worker / WebView 嵌套 **可用** | 模拟器能做,复杂图表用 Canvas 兜底 |
| 鉴权 | `wx.login()` → code 后端换 session(标准微信流程);`wx.getUserProfile()` 拿用户信息 | §3 鉴权设计明确 |
| 分享 | `wx.showShareMenu({ showShareItems: ['LuffachatFriends', 'LuffachatMoment'] })` + Page 级 `onShareAppMessage` 钩子返回卡片 | 新闻三联单分享路径已通 |
| 推送 | `wx.requestSubscribeMessage(tmplIds)` 订阅式模板消息(类微信) | 间隔复习用模板消息实现 |
| 后端通信 | `wx.request`(HTTPS)+ `wx.connectSocket`(WebSocket)+ `wx.uploadFile` | FastAPI 后端无障碍接入 |
| Canvas | `wx.createCanvasContext` + offscreen canvas + Path2D | IS-LM 沙盘 / 套息盯市动画 OK |
| H5 嵌套 | webview 组件 + 双向通信(`wx.miniProgram.postMessage`) | **救命路径**:极复杂模拟器可降级用 H5 实现 |
| 生物认证 | `wx.startSoterAuthentication`(指纹 / FaceID) | 用户回到自解释笔记的快速验证 |
| 性能 | iOS 端 **JIT 关闭**(JavaScriptCore 限制),Android V8 正常 | 复杂动画必须降级或 Canvas 实现 |

## 2. 前端技术栈

### 2.1 核心选型

| 维度 | 选择 | 原因 |
|---|---|---|
| 框架 | **Taro 3 + React 18 + TypeScript** | Taro 把 React 语法编译到 wx.* 小程序,SuperBox = 微信小程序架构,完美兼容。同 codebase 还能输出微信小程序版,扩大分发 |
| 构建 | **Taro CLI** | Taro 自带,无需自己配 |
| 路由 | **Taro 路由**(类小程序原生) | 不能用 React Router,因为不在浏览器环境。但 Taro 抽象了路由 API |
| 状态 | **Zustand + React Query** | Taro 完全兼容 React hooks,这一层不变 |
| 样式 | **Tailwind via taro-ui-plus 或 native-css** | Taro 4 起对 Tailwind 支持改善;退路是 SCSS module |
| 组件库 | **NutUI-React-Taro** 或 **Taroify** | 不能用 shadcn/ui(依赖 DOM)。NutUI/Taroify 是为小程序生态打造,暗色友好 |
| 动画 | Taro Animation API + CSS keyframes | iOS JIT 关闭,Framer Motion 太重,改用原生 transition + Canvas |
| 图表 | **Canvas 2D 自写**(模拟器关键图表)+ **echarts-for-taro**(简单看板图) | Recharts 不能用(SVG + DOM);D3 多数 API 需要 DOM,要 fork 计算层;Canvas 自写最稳 |
| 表单 | React Hook Form + Zod | Taro 兼容 |
| 国际化 | i18next | Taro 兼容 |
| 调试 | Taro **H5 模式 + SuperBox IDE 模拟器** | H5 模式开发体感接近浏览器,真机走 SuperBox IDE |

### 2.2 模拟器层架构

三章模拟器 + 后续可能扩展,需要一个**通用模拟器框架**(对应 §10 复用清单审计的"95% 框架复用"):

```
<SimulatorShell>
  <SnapshotBar />           // 历史快照按钮
  <SliderPanel sliders={...} />   // 滑块组
  <OutputPanel formula={fn} />    // 实时计算
  <ChallengeMode tasks={...} />   // §2.8 挑战
  <PredictionModal />       // §2.7 预测先行
  <SelfExplainPrompt />     // §2.9 自解释
  {children}                // 章节专属:动画 / SVG / Canvas
</SimulatorShell>
```

每章只需写:公式函数(≈ 10 行)+ 滑块配置(≈ 50 行)+ 历史快照数据(≈ 30 行)+ 章节专属可视化组件(≈ 200 行)。

**这个抽象的好处**:第 17、20 章的开发量从"重新做"降到"填表 + 写一个动画组件"。原型 v0 已经把这个壳的雏形写出来了,直接重构。

### 2.3 移动 / 桌面适配

- **mobile-first**:max-w-md 模拟手机,实际宽度 = 100vw 或 SuperBox 容器宽
- **桌面预览模式**:左侧手机壳模拟,右侧调试面板(教研用)
- **横屏处理**:模拟器在横屏下用 split layout

## 3. 后端与数据

### 3.1 选型

| 维度 | 选择 | 原因 |
|---|---|---|
| 语言 | **Python 3.12**(主)+ Node.js(辅,如需 Luffa SDK) | 因为后面 RAG / LLM / 数据 pipeline 都是 Python 主场 |
| Web 框架 | **FastAPI** | async 原生、OpenAPI 自动生成、Pydantic 类型友好 |
| ORM | **SQLModel**(FastAPI 作者出品)或 **SQLAlchemy 2** | SQLModel 学习曲线低,够用 |
| 数据库 | **Postgres 16** + **pgvector** | 关系型 + 向量库一体,运维少一个组件 |
| 缓存 | **Redis** | 因子数据 1 分钟级缓存、LLM 响应缓存 |
| 任务队列 | **Celery** 或 **dramatiq** | 数据 pipeline 调度 + 新闻三联单生成 |
| 鉴权 | **Authlib + JWT** | wx.login 拿 code → 后端换 JWT,标准流程 |

### 3.2 数据模型(核心表)

```
users                  -- 用户档案 + 元认知数据
chapters               -- 章节
concepts               -- 概念
misconceptions         -- 误区库(MM/IRP/ISLM 三章合 24 条)
quiz_questions         -- 题目(每题绑定 misconception 列表)
user_responses         -- 答题历史(用于诊断 + 间隔复习)
user_path_state        -- 路径分流状态
factor_snapshots       -- 因子时序数据(从 FRED 拉取)
historical_snapshots   -- 模拟器历史快照参数
news_trinity           -- 新闻三联单存档
self_explanations      -- 自解释文本 + AI 评分
ai_qa_logs             -- 答疑历史(用于改进 RAG)
embeddings (pgvector)  -- 教材讲义向量库
```

### 3.3 数据 pipeline

```
[FRED API]   ─┐
[OANDA FX]   ─┼─→ [Python ETL · Celery beat 每日触发]
[BIS / BoJ]  ─┘         │
                        ↓
              [Postgres factor_snapshots]
                        │
                        ↓
              [Redis 缓存 1min - 1day]
                        │
                        ↓
              [FastAPI /factors/* 端点]
                        │
                        ↓
              [前端 React Query 拉取]
```

- **MVP 数据源**:FRED(免费、关键)+ OANDA Free(汇率)+ 央行官网爬虫
- **付费源不进 MVP**:彭博 ≈ $2k/月,等用户量起来再升级
- **历史快照预计算**:第 14 章 4 个、第 17 章 5 个、第 20 章 5 个,直接 seed 进数据库,不实时算

## 4. RAG 知识库

### 4.1 内容来源(版权安全)

**绝不直接 ingest 米什金原文**(Pearson 版权)。RAG 知识库由这三类构成:

1. **自写讲义**:每章按 §13 误区库 + §1 学习目标重新组织,引用页码但不复制原文
2. **公开课讲义**:MIT Open / Coursera 已 CC 授权的相关材料
3. **官方文献**:Fed / BIS / IMF 工作论文(政府出版,公共领域或宽松授权)

每个 chunk 的元数据:

```json
{
  "chunk_id": "ch14-multiplier-001",
  "chapter": 14,
  "concepts": ["money_multiplier", "monetary_base"],
  "misconceptions": ["MM-03", "MM-04"],
  "level": "core" | "extension" | "challenge",
  "text": "...(自写文本)...",
  "page_refs": ["Mishkin §14.2"],
  "source_type": "self_written" | "ccby" | "public_domain"
}
```

### 4.2 选型

| 维度 | 选择 | 原因 |
|---|---|---|
| 向量库 | **pgvector** | Postgres 内,运维统一;到 100M chunk 才有性能问题,我们远远达不到 |
| Embedding | **Voyage 3 large** 或 OpenAI text-embedding-3-large | 中文表现 Voyage 略好 |
| 检索 | **混合检索**(BM25 + 向量,RRF 融合) | 单一向量在金融术语上召回差,关键词必须有 |
| 重排 | **Cohere Rerank 3** | 顶部 3-5 chunk 的精排,显著提升质量 |
| Chunking | 按概念切分,每段 300-500 tokens,带 100 token 重叠 | 教学内容自然有概念边界,不要用固定窗口切 |

### 4.3 检索流程

```
用户问题 "为什么 QE 没引发通胀?"
    │
    ├→ [意图分类] 是答疑/复习/导航?
    │
    ├→ [章节路由] 路径分流过滤(只搜该用户已学章节,避免剧透)
    │
    ├→ [混合检索] BM25 top-20 + 向量 top-20 → RRF 合并
    │
    ├→ [Rerank] Cohere Rerank → top-5
    │
    ├→ [上下文组装]
    │     - 5 个 chunk
    │     - 该用户最近 30 天因子数据快照
    │     - 该用户答错过的题 + 触发的误区代码
    │
    └→ [LLM 生成]
          system: 答疑助教 prompt 模板(02 §6.2)
          → 流式响应 → 前端
```

### 4.4 评估

每周用 50 道"金标问题"(教研写)跑一遍 RAG 系统,人工评分。指标:

- **命中率**:top-5 是否包含正确章节
- **Hallucination 率**:答案是否引用了不存在的页码或事实
- **教学价值**:是否触达误区、是否给出"模拟器跳转"等下一步动作

## 5. LLM 接入

| 用途 | 模型 | 配置 |
|---|---|---|
| 答疑(高频) | **Claude Sonnet 4.6** | streaming + prompt caching(系统 prompt + RAG context 缓存) |
| 自解释评分(中频) | **Claude Sonnet 4.6** | structured output(JSON 评分) |
| 教别人评估(低频) | **Claude Opus 4.7** | 非 streaming,质量优先 |
| 新闻三联单生成 | **Claude Opus 4.7** | 离线批量,人工审核必须 |

**成本估算**(MVP 1000 DAU):

- 答疑:平均 5 次/用户/章 × 3 章 × 1k tokens/次 ≈ 15k tokens/DAU/月,千用户 ≈ 15M tokens/月
- 缓存命中后真实计费 ≈ 30-50%,Sonnet 输入 $3/M、输出 $15/M
- 月 LLM 成本估算:$200-400 / 1k DAU

**Prompt caching 必做**:RAG 召回的章节内容、用户档案、系统 prompt 全部走 cache。命中率达到 70% 后总成本可降到 $100 / 1k DAU。

## 6. Luffa SDK 接入(v1 · 基于已确认 API)

抽象层比 v0 设想轻得多——`wx.*` 已经是统一接口,不需要再包一层 Adapter,直接封装成 service:

```typescript
// services/luffa.ts
import Taro from '@tarojs/taro'

export const luffa = {
  // ✅ 鉴权(已确认)
  async login() {
    const { code } = await Taro.login()
    return code  // 后端拿这个 code 换 sessionKey
  },
  async getUserProfile() {
    return await Taro.getUserProfile({ desc: '用于个性化学习路径分流' })
  },

  // ✅ 分享(已确认)
  enableShare() {
    Taro.showShareMenu({
      withShareTicket: true,
      showShareItems: ['LuffachatFriends', 'LuffachatMoment']
    })
  },
  // 卡片内容由 Page 级 onShareAppMessage 钩子返回
  buildShareCard(card: { title: string; imageUrl: string; path: string }) {
    return {
      title: card.title,
      imageUrl: card.imageUrl,
      path: card.path  // deeplink 进对应章节
    }
  },

  // ✅ 推送(已确认)
  async subscribePush(templateIds: string[]) {
    return await Taro.requestSubscribeMessage({ tmplIds: templateIds })
  },

  // ✅ 网络(已确认,用 Taro 包装)
  async apiCall<T>(endpoint: string, payload?: any): Promise<T> {
    const { data } = await Taro.request<T>({
      url: `${API_BASE}${endpoint}`,
      data: payload,
      method: payload ? 'POST' : 'GET',
      header: { 'Authorization': `Bearer ${getSession()}` }
    })
    return data
  },

  // ⚠️ 待 Luffa BD 确认(占位,目前抛错)
  async joinChannel(channelId: string) {
    throw new Error('待 Q-A 答复')
  }
}
```

**关键判断**:Q-A(Channel)未确认前,产品功能可以**降级实现**:

- Channel 通过 → 完整做"读书会""1v1 PK"等深度社交
- Channel 不通 → 只做"分享给好友"(已确认),延后到 v2

这个降级路径意味着:**MVP 不会被技术问题卡死**,Q-A 答复影响的是 v1.5 / v2 功能扩张。

## 7. 部署架构

| 组件 | 选型 | 原因 |
|---|---|---|
| 前端托管 | **Cloudflare Pages** | 免费额度大、CDN 自带、和 Workers 集成方便 |
| 后端 | **Fly.io** 或 **Railway** | 部署简单、按量计费、Postgres 自动管理 |
| Postgres + pgvector | **Supabase** 或 **Neon** | 托管 + 自带 pgvector,运维零成本 |
| Redis | **Upstash** | serverless redis,免费额度够 MVP |
| 任务队列 | Fly.io Workers 或自建 | Celery beat + worker |
| 监控 | **Sentry**(错误)+ **PostHog**(产品分析) | PostHog 对学习产品的 funnel 分析特别好用 |
| LLM 网关 | **直连 Anthropic API** | 不引入额外抽象层 |
| 文件 / 静态资源 | **Cloudflare R2** | S3 兼容,出口免费 |

**MVP 月成本估算(1000 DAU)**:

| 项 | 成本 |
|---|---|
| Cloudflare Pages + R2 | $0-10 |
| Fly.io 后端 + worker | $20-50 |
| Supabase Pro | $25 |
| Upstash Redis | $10 |
| LLM(见 §5) | $200-400 |
| Sentry + PostHog | $0-30 |
| FRED / OANDA(免费层) | $0 |
| **合计** | **$255-525 / 月** |

加上一台教研 staging 环境,约 $400-700/月,够用到 5000 DAU。
